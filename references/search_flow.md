# AI 问答详细流程（语义增强搜索）

> 本文档包含 AI 问答的完整搜索流程、Rerank 策略、答案格式和死条目处理。
> 概述和触发条件见 SKILL.md → 第 9 节。
> API 调用使用 [api_helper.py](api_helper.py) 中的通用封装（`yuque_get`、`parallel_get` 等）。

## 搜索流程

```
用户提问
    ↓
查搜索缓存（`search_cache.json`，TTL=`cache_ttl_minutes` 分钟）
    ↓
         命中且未过期？
       ╱              ╲
     是                否
    ╱                   ╲
直接返回          LLM Query 扩展 + 同义词扩展 + 实体识别
（跳过搜索，      （查 synonym_map + 识别 @实体: 词 + LLM 补漏）
 减少 LLM 调用）       ↓
                  并行搜索 index_master_book
                   （`ThreadPoolExecutor`，并发数 ≤ 5）
                       ↓
获取子文档列表 → 合并索引条目（按源文档ID去重）
    ↓
    提取条目的「实体」和「关系」字段（存入上下文，供 LLM 回答时优先使用）
    ↓
                    结果数 = 0？
                  ╱           ╲
               是               否
              ╱                  ╲
    补充原生搜索              索引命中
   （同 scope 搜索，          ↓
    按源文档ID去重合并） 标题+摘要关键词匹配粗排
              ╲              （候选 > 10 时取前 10）
               ╲               ╱
               统一 LLM Rerank（对候选源文档打分重排）
                    ↓
           并行获取 Top K 全文（`ThreadPoolExecutor`，并发数 ≤ 5）
                    ↓
           分段处理 + LLM 生成回答（实体/关系数据作为回答上下文注入）
```

## 索引库设计（关键词索引 + 总文档）

- 每个关键词对应一个**总文档** + 多个**子文档**
- 总文档标题：`[索引] {关键词}`，存储元信息（JSON 格式）
- 子文档标题：`[索引] {关键词} (1)`、`[索引] {关键词} (2)`...
- 子文档存储具体索引条目，接近 200kb 时新建下一个子文档
- 搜索时匹配总文档标题，从总文档获取子文档列表

## 同义词索引设计

- 索引构建时：LLM 提取关键词并扩展同义词，每个同义词都创建索引文档
- 例如：「Python安装」文档 → 同时索引到 `Python`、`py`、`python3`、`安装`、`install` 等条目
- 搜索时：LLM 扩展用户关键词，覆盖索引阶段遗漏的同义词

## 搜索参数

- `candidates_limit`: 候选数，默认 20
- `top_k`: 获取全文数，默认 5
- `segment_length`: 分段长度，默认 2000
- `cache_ttl_minutes`: 缓存有效期，默认 30

## 搜索缓存

配置文件 `search_cache` 指定缓存位置（默认 `~/.openclaw/workspace/utils/yuque/search_cache.json`）。`cache_ttl_minutes` 控制有效期。

```json
{
  "md5(用户问题)": {
    "answer": "根据你的语雀笔记...",
    "sources": ["doc_id": 123, ...],
    "timestamp": "2026-05-03T11:00:00+08:00"
  }
}
```

缓存命中 → 直接返回；缓存过期或未命中 → 走完整搜索流程 → 写入缓存。

## 实体增强搜索

索引条目中的「实体」和「关系」字段会作为额外上下文注入 LLM 回答。LLM 优先用结构化关系数据直接回答，不足时结合原文。

### 工作原理

```
用户问「亿级流量系统用了什么技术」
  → 搜索结果命中条目，提取到：
     实体: @实体:Redis(中间件), @实体:Elasticsearch(中间件)
     关系: Redis→用于→亿级流量系统 | ES→用于→亿级流量系统
  → 注入 LLM prompt（与原文分段并列）
  → LLM 优先用关系数据回答：「根据你的笔记，亿级流量系统使用了 Redis 和 Elasticsearch」
  → 无需读长文档找技术栈，结构化数据直接命中
```

### 降级策略

实体/关系数据不足以回答时（如用户问的是实现细节而非关系），LLM 自动从原文分段中获取。不需要前置判断——LLM 拿到两份上下文（关系数据 + 原文分段），自己选最合适的。

## 搜索阶段同义词扩展

**优先查缓存**：读取配置文件 `synonym_map` 路径（默认 `~/.openclaw/workspace/utils/yuque/synonym_map.json`），对提取的关键词逐一查表。

```
用户问「py 怎么装」

→ LLM 提取关键词：py, 安装
→ 查 synonym_map：
  - py → 命中：[Python, python3, py3]
  - 安装 → 命中：[install, 环境配置, 环境搭建]
→ 合并得到搜索词：py, Python, python3, py3, 安装, install, 环境配置, 环境搭建

若某个词缓存未命中（生僻词），单独调 LLM 补充同义词并写入缓存。

→ 搜索索引文档：
  py ← 命中
  Python ← 命中
  python3 ← 命中
  安装 ← 命中
  install ← 命中
  环境配置 ← 命中

→ 合并索引条目，按源文档ID去重
→ Rerank → 取 Top K 全文
```

## 同义词类型

- 缩写/简写：Python → py, JavaScript → js, Kubernetes → k8s
- 中英文互译：安装 → install, 配置 → config, 线程 → thread
- 近义词：安装 → 部署 → 环境搭建, 数据库 → DB → 存储
- 大小写变体：Python → python, JVM → jvm
- 专业术语变体：线程池 → ThreadPool, 并发 → concurrency

## 搜索步骤详解

1. 读取配置获取 `index_master_book`
2. LLM 扩展用户问题为多个搜索关键词，同时识别命名实体生成 `@实体:xxx` 形式的搜索词
3. **同义词扩展**：读取 `synonym_map.json` 查表 → 命中直接取；未命中的生僻词调 LLM 补充并写入缓存。`@实体:` 关键词跳过同义词扩展（实体名本身已唯一）
4. **并行搜索索引库**：对每个关键词（含同义词），调用 `/api/v2/search?q={关键词}&type=doc&scope={namespace}` 搜索索引总库（`ThreadPoolExecutor`，并发数 ≤ 5）。首条精确匹配标题即视为命中。

> ⚠️ **不要用** `GET /repos/{book_id}/docs` 做搜索，此端点无 `q` 参数。必须用 `/api/v2/search`。
5. 若首条标题精确匹配关键词，获取该索引文档内容（`raw=1`）→ 解析 `sources` 获取源文档列表
6. 若无精确匹配（首条标题≠关键词），尝试相近关键词重新搜索
7. 合并所有子文档内容，**按源文档ID去重**索引条目
8. **索引降级兜底**：合并去重后若结果数为 0（索引未构建或未覆盖该话题）→ 降级补搜：
   - 使用语雀原生搜索 API，**不传 `scope`**（搜全部），用 LLM 扩展后的关键词**并行搜索**（`ThreadPoolExecutor`，并发数 ≤ 5）
   - 原生搜索结果按源文档ID去重，与索引结果合并（此时索引为空，实际全是原生结果）
   - 原生搜索返回的 `summary` 作为摘要参与后续 Rerank
9. **粗排**（候选 > 10 时执行）：用 LLM 扩展后的关键词与候选的标题 + 摘要做关键词匹配度评分，取前 10 篇进入 Rerank。候选 ≤ 10 时跳过，直接进入 Rerank。
10. LLM 对候选源文档 Rerank 打分，按与用户问题的**语义相关性从高到低排序**（相关性相同时优先时间较新的文档）
11. 取 Top K 源文档，**并行获取全文**（语雀 API QPS=100，并发数 ≤ 5）
   - **200 OK** → 正常，进入分段处理
   - **404/410** → 文档已删除：跳过，顺延取下一个候选；记录死条目（源文档ID + 标题 + 链接 + 此条目的关键词列表）→ 追加到状态文件 `dead_entries`（去重：同 doc_id 不重复添加）
   - **429** → 检查 `X-RateLimit-Remaining` 响应头：
     - `remaining = 0` → 触及 **5000/h 限制**，立即暂停，汇报：「⏳ 已达小时频率上限，已跳过后续候选。整点后重新搜索即可。」
     - `remaining > 0` → 触及 **100/s 限制**，等待 1s 重试（最多 3 次），3 次均失败则顺延
   - **5xx / 超时** → 重试 1 次，仍失败则顺延取下一个候选
12. 分段处理（按 Markdown 标题 `##`/`###` 切分为逻辑段，段 > 2000 字时按段落 `\n\n` 拆分，仍超则按字数兜底；每段带章节标题喂 LLM）。**将匹配条目的实体/关系数据作为额外上下文一起注入**，LLM 优先用结构化关系回答，不依赖前置判断。
13. LLM 生成回答，标注来源
14. **若使用了原生搜索兜底**：在回答末尾追加提示
15. **若有死条目**：追加到状态文件 `dead_entries`（按 doc_id 去重）。若 `dead_entries` 条数 > 配置文件 `dead_entries_threshold`（默认 10），在回答末尾提示清理

## 回答格式

```markdown
根据你的语雀笔记，找到以下相关内容：

📝 **文档标题**
来源：知识库名称
链接：https://www.yuque.com/xxx/xxx

...AI 总结内容...

---
共参考 N 篇文档
```

若通过原生搜索兜底获取结果，末尾追加：

```markdown
💡 当前为原生搜索结果，语义精度有限。构建索引后可获得更精准的 AI 搜索体验。
```

## 死条目处理

### 清理提示格式

死条目积累到阈值时（条数 > 配置文件 `dead_entries_threshold`，默认 10），在回答末尾追加：

```markdown
---
⚠️ 已积累 12 条死索引条目（源文档已删除），回复「清理死索引」一键移除。
```

单次搜索命中死条目但未达阈值时，仅追加到状态文件不提示。

### 历史死条目查看

用户可随时查询：`dead_entries` 字段，格式化展示：

```markdown
⚠️ 当前累计 8 条死索引条目：

1. 📄 **Python 安装指南** (doc: 205935051)
   索引词：Python, py, 安装, install（等 9 个关键词）
   链接：https://www.yuque.com/yehuoshun/wwqac0/abc123

2. 📄 **旧版 README** (doc: 301234567)
   索引词：README, 说明, 文档
   链接：https://www.yuque.com/yehuoshun/wwqac0/def456
```

### 清理流程

用户说「清理死索引」时，**批量清理状态文件中的所有死条目**：

1. 读取 `dead_entries` 列表，**预估总 API 调用次数**（每关键词 ≈ 1 次搜索 + 1 次读取 + 1 次 PUT，即每个死条目的每个关键词 ≈ 3 次调用）。若预估 > 200 次调用，先汇报估算并向用户确认
2. 按死条目逐条处理（每条 = 一个源文档的全部关键词集合）。**每处理完一条死条目后检查上次 API 调用返回的 `X-RateLimit-Remaining`**：`remaining = 0` 时暂停，汇报用户整点后重新触发；`remaining < 100` 时等待 2 秒后继续
3. 对每个死条目取出 `keywords` 列表，逐个关键词处理：
   - 在 `index_master_book` 中定位 `[索引] {关键词}` 总文档 → 遍历子文档找到匹配该 `doc_id` 的条目并删除
   - PUT 更新有变动的子文档
4. 若预估调用 > 100 次，分阶段汇报进度（如「已清理 3/8 个源文档…」）
5. 清空 `dead_entries`，汇报：清理了几个源文档，涉及多少个子文档，总 API 调用次数
